Middleware Pattern (Interception)
The theory behind Middleware is "intercepting" a request before it reaches your logic.
The Cache Middleware: This is a Proxy Pattern. Instead of hitting your heavy computeReport logic every time, the middleware checks a "store" (like your TTL-indexed MongoDB or a local variable) first. It saves CPU cycles and prevents you from hitting GitHub's rate limits by serving a "snapshot" of the data.
The CORS Lock: This is your primary Network Security layer. By restricting origins, you’re preventing "Cross-Site Request Forgery" and stopping other people from "stealing" your backend API to power their own websites.


Centralized Error Handling
Instead of having try/catch blocks that send different types of responses everywhere, a Global Error Handler creates Consistency.
Theory: This is the Boundary Pattern. It ensures that no matter where an error happens (database, GitHub API, or a syntax typo), the frontend always receives a predictable JSON object (like { error: "Message", status: 500 }) rather than a raw HTML crash page or a leaked stack trace.
